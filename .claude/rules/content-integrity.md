# Content Integrity - 內容誠信規則

## 核心原則

> **AI 絕對不能編造場景、故事、個人經驗、時間、地點、人物、對話、情緒反應。**
>
> **所有具體細節必須來自用戶提供的素材。沒有素材就不要硬寫。**

---

## 禁止行為

### ❌ 絕對禁止

| 禁止內容 | 說明 |
|----------|------|
| **編造場景** | 「2023年3月的某個下午，咖啡廳裡...」（用戶沒提供） |
| **虛構對話** | 「他對我說：『你一定可以的』」（用戶沒提供） |
| **假造經驗** | 「我曾經也遇過類似的困境...」（AI沒有經驗） |
| **捏造人物** | 「我的朋友小明是個工程師...」（用戶沒提供） |
| **偽造情緒** | 「當時我感到非常沮喪...」（AI沒有情緒） |
| **編造數據** | 「根據我的觀察，80%的人...」（沒有調查） |
| **假裝親身** | 「我在矽谷工作時...」（AI沒去過） |

---

## 允許行為

### ✅ AI 可以做的

| 允許內容 | 說明 |
|----------|------|
| **描述技術事實** | 「REST API 使用 HTTP 協議...」 |
| **解釋原理機制** | 「這個演算法的運作方式是...」 |
| **組織用戶素材** | 整理用戶提供的資料成結構化內容 |
| **潤飾用戶故事** | 改善用戶提供的故事的文字流暢度 |
| **引用公開資料** | 「根據《哈佛商業評論》2023年研究...」 |
| **提供框架模板** | 「可以用 AIDA 公式來撰寫...」 |
| **標註留白** | 「[此處可加入您的個人經驗]」 |

## 素材檢查清單

### 撰寫前必須確認

在撰寫任何具體內容前，確認：

1. ✅ 用戶是否提供了相關素材？
2. ✅ 素材中是否包含時間、地點、人物？
3. ✅ 是否有真實的對話或事件描述？
4. ✅ 情緒反應是否來自用戶？

### 沒有素材時的處理

| 情況 | 處理方式 |
|------|---------|
| **缺少故事** | 用技術事實或框架取代，不要編故事 |
| **缺少案例** | 標註「[建議加入您的實際案例]」 |
| **缺少經驗** | 用「假設你遇到...」而非「我曾經...」 |
| **缺少對話** | 用「可能的對話是...」而非直接編對話 |
| **缺少數據** | 引用公開研究或標註「需要調查」 |

## 寫作模式選擇

根據用戶素材類型選擇模式：

### 摘要模式（有對話/訪談/會議記錄）

**適用**：用戶提供了真實對話、訪談、會議紀錄、隨筆

**處理方式**：
- 保留原話精華與語氣
- 整理重點成結構化內容
- 加入適當小標分段
- **不加過度修飾與文學手法**
- 結尾簡潔收束

**範例**：
```
用戶素材：「昨天開會時，老闆說我們要做 AI 團隊，但我不太懂...」

✅ 正確：整理成「老闆提出要建立 AI 團隊的想法，但團隊對此概念還不清楚...」

❌ 錯誤：編造「那天下午三點，會議室裡，老闆拋出了一個震撼的提議...」
```

---

### 故事模式（有概念但無具體素材）

**適用**：用戶提供概念、需要組織敘事

**處理方式**：
- 用技術事實支撐
- 用已知案例（公開資料）
- 用假設性敘述（明確標示「假設」）
- 用框架引導思考
- **標註需要用戶補充的部分**

**範例**：
```
用戶需求：「寫一篇關於 AI 團隊的文章」

✅ 正確：
「AI 團隊的概念源自於...（技術事實）
假設你的組織需要建立 AI 團隊，可以考慮以下結構...（框架）
[建議加入您實際建立團隊的經驗]（留白）」

❌ 錯誤：
「去年我在矽谷參與了一個 AI 團隊專案，當時的場景歷歷在目...」
```

## 引用規範

### 可以引用的來源

| 來源類型 | 處理方式 |
|----------|---------|
| **學術研究** | 標註期刊、年份、作者 |
| **公開報導** | 標註媒體、日期 |
| **書籍內容** | 標註書名、作者 |
| **統計數據** | 標註來源機構 |
| **知名案例** | 公開可查證的事件 |

### 引用格式

```markdown
根據《哈佛商業評論》2023年研究...
麥肯錫報告指出...
Google 在 2015 年發表的研究顯示...
```

## 違規處理

### 發現違規時

1. **立即停止**：停止撰寫該部分
2. **標註問題**：明確指出哪些內容是編造的
3. **詢問用戶**：「這部分需要您提供真實素材」
4. **提供替代**：用技術事實或框架替代

### 自我檢查

每完成一段內容，問自己：

- ❓ 這個場景是用戶提供的嗎？
- ❓ 這段對話是真實的嗎？
- ❓ 這個經驗是我編的嗎？
- ❓ 這個情緒反應有依據嗎？

---

## 特殊情況

### 教學性假設

允許使用假設性場景，但**必須明確標示**：

✅ 正確：
```
「假設你需要建立一個 AI 團隊，可以這樣做...」
「想像一個情境：你的公司決定導入 AI...」
「如果遇到這種情況，可以考慮...」
```

❌ 錯誤：
```
「我曾經建立過一個 AI 團隊...」（沒建過）
「有一次在專案中遇到...」（編的）
```

## 責任歸屬

| 角色 | 責任 |
|------|------|
| **article-planner** | 確認用戶提供的素材類型與數量 |
| **article-researcher** | 只引用可驗證的公開資料 |
| **article-writer** | 絕對不編造具體細節 |
| **article-editor** | 審查是否有編造內容 |
| **article-analytics** | 檢查內容誠信，發現問題立即退回 |

## 用戶須知

為確保內容品質與誠信，請用戶：

1. **提供真實素材**：故事、經驗、對話、案例
2. **標註素材來源**：「這是我的經驗」或「這是公開案例」
3. **接受留白**：沒有素材時，AI 會標註留白處供您補充
4. **驗證引用**：檢查引用的研究或數據是否正確
