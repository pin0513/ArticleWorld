# Article Structure Guide - 文章架構指南

## 核心概念

> **架構不等於風格。架構服務於寫作目的。**

- **風格/口吻**：how you say it（怎麼說）
- **架構/結構**：what structure you use（用什麼結構）

**本 skill 專注於架構，不涉及風格。**

---

## 四大核心架構

根據寫作目的，文章架構分為 4 種：

| 寫作目的 | 對應架構 | 讀者期望收穫 |
|---------|---------|-------------|
| 證明某個方法有效 | 實證論述架構 | 學到一個可執行的方法 |
| 分析某個現象 | 現象解析架構 | 理解一個趨勢或變化 |
| 分享個人經驗 | 經驗敘事架構 | 從別人經驗中學習 |
| 引發思考 | 思辨探討架構 | 獲得新的思考角度 |

---

## 架構 1：實證論述架構

### 適用情境

- 你有一個方法/工具/框架要推薦
- 你想證明這個方法有效
- 讀者看完後能立即採用

### 標準結構

```markdown
# 標題（明確說明方法）

## 1. 問題定義 (10%)
- 當前痛點是什麼
- 為什麼現有方案不夠好
- 這個問題的規模/影響

## 2. 方案說明 (30%)
- 你的解決方案是什麼
- 為什麼這樣設計
- 與現有方案的差異
- 核心機制解釋

## 3. 實證驗證 (40%)
- 測試數據（即使是小規模）
- 對比基準（之前 vs 之後）
- 成功案例
- 失敗案例與調整
- 適用場景與限制

## 4. 應用指南 (20%)
- 讀者如何採用（3-5個步驟）
- 常見坑與注意事項
- 進階資源
```

### 必要元素檢查

- [ ] 有明確的問題定義
- [ ] 方案說明佔全文 30% 以上
- [ ] 有實際數據或案例
- [ ] 有對比基準（改善前後）
- [ ] 有失敗案例或限制說明
- [ ] 有可執行的應用步驟

### 禁止元素

- ❌ 感性開場（「你還記得嗎」）
- ❌ 哲學思辨段落
- ❌ 雞湯式結尾
- ❌ 沒有數據支撐的效率宣稱
- ❌ 超過 2 段的個人故事

### 範例標題

- 「如何用 v0-v3 架構降低 AI 團隊成本 30%」
- 「實測：用 CODEMAPS 索引讓 AI 理解代碼速度提升 2 倍」
- 「團隊設計模式演進：從單人到梯隊的完整指南」

## 架構 2：現象解析架構

### 適用情境

- 你觀察到一個行業趨勢或變化
- 你想分析這個現象背後的原因
- 讀者看完後能理解這個趨勢

### 標準結構

```markdown
# 標題（描述現象）

## 1. 現象描述 (15%)
- 具體的變化趨勢
- 數據或案例佐證
- 對比過去與現在

## 2. 原因分析 (35%)
- 為什麼會發生這個變化
- 底層驅動力是什麼
- 技術/社會/經濟因素
- 歷史脈絡

## 3. 影響評估 (30%)
- 對誰有影響
- 好的影響 vs 壞的影響
- 短期影響 vs 長期影響
- 案例說明

## 4. 應對建議 (20%)
- 個人層面可以做什麼
- 組織層面可以做什麼
- 具體行動步驟
```

### 必要元素檢查

- [ ] 現象描述有具體數據或案例
- [ ] 原因分析有邏輯鏈
- [ ] 影響評估具體（誰受影響、如何影響）
- [ ] 應對建議可執行
- [ ] 有對比（過去 vs 現在）

### 禁止元素

- ❌ 只有現象描述沒有分析
- ❌ 分析過於抽象（「多方面因素」）
- ❌ 只有分析沒有建議
- ❌ 建議過於空泛（「需要適應變化」）

### 範例標題

- 「工程師角色消亡：從個人貢獻者到團隊設計者的轉型」
- 「AI 時代的職位邊界消融：為什麼全端工程師不夠了」
- 「軟體產業的範式轉移：代碼不再是主要產出」

## 架構 3：經驗敘事架構

### 適用情境

- 你有一段真實的經歷想分享
- 這段經歷有學習價值
- 讀者可以從你的經驗中學習

### 標準結構

```markdown
# 標題（明確說明經驗主題）

## 1. 情境設定 (10%)
- 時間、地點、背景
- 為什麼做這件事
- 初始狀態

## 2. 過程記錄 (40%)
- 具體發生了什麼（時間線）
- 遇到什麼問題
- 如何嘗試解決
- 失敗的嘗試
- 最終解法

## 3. 反思提煉 (30%)
- 學到什麼
- 如果重來會怎麼做
- 哪些經驗可複製
- 哪些是特殊情況

## 4. 給讀者的啟發 (20%)
- 哪些可以參考
- 哪些要避免
- 如何應用到自己的情境
```

### 必要元素檢查

- [ ] 有具體的時間、地點、人物
- [ ] 有失敗經驗（不是只有成功）
- [ ] 有具體的問題解決過程
- [ ] 反思有深度（不是流水帳）
- [ ] 啟發可執行

### 禁止元素

- ❌ 泛泛而談（「我們常常遇到...」）
- ❌ 假經驗（AI 編造的）
- ❌ 只有成功沒有失敗
- ❌ 沒有反思的流水帳
- ❌ 教條式總結（「所以我們要...」）

### 範例標題

- 「我如何用 3 個月建立 AI 開發團隊：從 0 到 10 人的真實記錄」
- 「第一次導入 Claude Code：5 個失敗教訓與 3 個成功關鍵」
- 「團隊架構重構實戰：v1 到 v3 的踩坑與收穫」

## 架構 4：思辨探討架構

### 適用情境

- 你對某個議題有深度思考
- 這個議題沒有標準答案
- 你想引發讀者思考

### 標準結構

```markdown
# 標題（提出核心問題）

## 1. 核心問題 (10%)
- 拋出一個沒有標準答案的問題
- 為什麼這個問題值得思考
- 這個問題的背景

## 2. 多角度探討 (50%)
- 觀點 A 及其理由
- 觀點 B 及其理由
- 兩者的矛盾點
- 現實中的案例
- 不同情境下的答案

## 3. 個人立場 (20%)
- 我傾向哪邊
- 為什麼（個人經驗或思考）
- 我還不確定的部分
- 我的立場有什麼限制

## 4. 開放結尾 (20%)
- 留給讀者的問題
- 歡迎討論的點
- 未來可能的變化
```

### 必要元素檢查

- [ ] 問題有思考價值（不是顯而易見的答案）
- [ ] 探討有深度（不是表面辯論）
- [ ] 個人立場明確（不是假中立）
- [ ] 有具體案例支撐觀點
- [ ] 承認不確定（不是假裝什麼都懂）

### 禁止元素

- ❌ 假中立（「每個人可能有不同看法」）
- ❌ 沒有個人觀點的純理論探討
- ❌ 強行結論（「所以答案是...」）
- ❌ 兩面都說但沒深度
- ❌ 正能量強行收尾

### 範例標題

- 「團隊是人的組成，還是職位的組成？」
- 「當 AI 能寫代碼，工程師的價值還剩什麼？」
- 「軟體成為副產品：思維躍遷還是自我安慰？」

## 架構混用判定

### 什麼時候會架構混用？

```
常見錯誤案例：

1. 想證明方法有效（實證論述）
   但用了感性開場（經驗敘事）
   → 讀者不知道你要幹嘛

2. 想分析現象（現象解析）
   但加入大量個人故事（經驗敘事）
   → 分析深度不夠

3. 想分享經驗（經驗敘事）
   但寫成方法論（實證論述）
   → 缺少具體情境，變成空泛建議

4. 想引發思考（思辨探討）
   但強行給結論（實證論述）
   → 失去開放性
```

### 如何避免混用？

1. **寫作前明確目的**：用一句話說明這篇文章要達成什麼
2. **選定一個主架構**：不要想「兼顧」多種目的
3. **次要架構不超過 20%**：可以有其他架構元素，但不能喧賓奪主

## 架構審核清單

### 審核步驟

```
Step 1: 識別寫作目的
從內容判斷作者想證明/分析/分享/探討什麼

Step 2: 檢查架構匹配
實際使用的架構是否服務於寫作目的

Step 3: 結構完整性檢查
該有的段落是否齊全，比例是否合理

Step 4: 內容密度檢查
每種架構的必要元素是否具備
```

### 審核輸出格式

```markdown
## 架構審核結果

### 識別的寫作目的
[證明方法/分析現象/分享經驗/引發思考]

### 應該使用的架構
[實證論述/現象解析/經驗敘事/思辨探討]

### 實際使用的架構
[描述實際結構]

### 架構匹配度
✅ 完全匹配 / ⚠️ 部分混用 / ❌ 架構錯位

### 問題點
1. [具體問題]
2. [...]

### 重構建議
[如何調整架構]
```

## 使用指南

### 寫作前（article-planner）

使用「架構選擇決策樹」，在企劃階段就確定架構：

```markdown
## 架構選擇

寫作目的：證明團隊設計模式（v0-v3）有效
選定架構：實證論述架構

必要元素檢查：
- [ ] 問題定義
- [ ] 方案說明（含 v0-v3 演進）
- [ ] 實測數據
- [ ] 應用指南
```

### 審稿時（article-editor）

使用「架構審核清單」，檢查架構是否正確：

```markdown
## 架構審核

目的：證明方法有效
架構：實證論述
匹配度：❌ 架構錯位

問題：
1. 開頭用感性引入（經驗敘事風格）
2. 缺少實測數據
3. 結尾用哲學思辨收尾

建議：重寫，用實證論述架構
```

## 常見問題

### Q1: 可以混用架構嗎？

可以，但有限制：
- **主架構佔 80%**：明確一個主要架構
- **次要元素不超過 20%**：其他架構元素點綴即可

範例：實證論述為主，開頭用 1 段經驗引入（不超過 10%）

### Q2: 架構 vs 風格的關係？

- **架構**：文章的骨架（結構）
- **風格**：文章的肌肉（語言風格）

同一個架構（如實證論述），可以用不同風格：
- 商業風格：數據導向、專業術語
- 教練風格：對話式、引導思考
- 故事風格：案例豐富、畫面感強

參考 `skills/writing-styles/` 了解風格系統

### Q3: 如何處理用戶混亂的需求？

如果用戶的需求本身就混亂（又要證明方法，又要分析現象，又要講故事）：

1. **拆成多篇**：每篇一個明確目的
2. **選擇主要目的**：問用戶「最想達成什麼」
3. **系列文章**：第一篇現象解析，第二篇實證論述
